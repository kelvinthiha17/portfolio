// STM32 · Embedded C · FreeRTOS

STM32 Embedded Sensor & Control System

Embedded C STM32CubeIDE FreeRTOS PWM Interrupt-Driven

A multi-peripheral embedded firmware project built on an STM32 (Cortex-M0+) microcontroller in STM32CubeIDE. The system integrates GPIO-timed sensor interfacing, hardware PWM generation for servo motor control (sweeping a servo 0°–180° via TIM3), and interrupt-driven timer input capture for precision pulse-width signal timing.

Going multitasking

On top of the base firmware, I integrated X-CUBE-FREERTOS via CubeMX's Software Packs Manager and built a two-task system on the CMSIS-RTOS2 interface: an onboard LED task (PA5, 500ms) and a second, independently scheduled task driving an external LED on a breadboard (PA7, through a 1kΩ resistor, 200ms) — both toggling on their own osDelay() cycles with no blocking, proving out real RTOS scheduling rather than a single-loop illusion of it.

Real-time UART telemetry: alongside the RTOS tasks, the system streams live debug output over a serial console — timer states, capture values, and task status — so behavior can be observed as it happens rather than reconstructed after the fact.

The debugging that actually taught me something

The harder lesson came from a separate ultrasonic distance sensor integration attempt, which surfaced two real HAL-level bugs worth documenting. First, UART wasn't initializing correctly because USART1 needed BSP_COM_Init() paired with hcom_uart[COM1] — not the manually generated MX_USART1_UART_Init() CubeMX had produced. Second, HAL_TIM_IC_Start_IT() kept failing because HAL_TIM_IC_Stop_IT() was never being called on timeout, leaving the HAL state machine permanently stuck mid-capture.

Fixing both restored reliable UART communication and timer-interrupt behavior — the kind of register-level debugging that doesn't show up in a tutorial, but is exactly what separates "the code compiles" from "the peripheral actually works."

Technologies: STM32CubeIDE, STM32CubeMX, HAL drivers, FreeRTOS (CMSIS-RTOS2), PWM, timer input capture, interrupt-driven programming, UART.
