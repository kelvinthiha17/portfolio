// Verilog · RTL Design · CPU Architecture

MiniRISC16 Processor

Verilog RTL Design CPU Architecture Icarus Verilog GTKWave 16-bit

A fully functional 16-bit RISC processor designed from scratch in Verilog. The architecture follows a classic single-cycle datapath — each instruction completes in one clock cycle, flowing through the program counter, instruction memory, decoder, register file, and ALU. All modules were individually unit-tested and verified together using Icarus Verilog simulation with GTKWave waveform analysis.

Program Counter
pc.v
Resets to 0x0000 on reset, increments by 1 each clock edge. Drives the instruction fetch address.
Instruction Memory
imem.v
256 × 16-bit ROM. Combinationally outputs the instruction at the address from the PC. Pre-loaded with a test program.
Instruction Decoder
decoder.v
Splits the 16-bit instruction word into opcode, rd, rs, rt, and imm fields using bit slicing.
Register File
regfile8x16.v
8 registers × 16 bits. Two combinational read ports (rs, rt). One synchronous write port (rd) — writes on positive clock edge.
ALU
alu16.v
16-bit combinational ALU. Supports ADD, SUB, AND, OR, XOR selected by a 3-bit opcode. Includes a zero flag for future branch instructions.
Top-Level CPU
cpu.v
Wires all five modules together. ALU result feeds directly back into the register file write port — single-cycle, no pipeline stalls.

// instruction format

All instructions are 16 bits wide. The high 4 bits are the opcode; register fields are 3 bits each.

[2:0]
opcode [15:12] rd [11:9] — destination register rs [8:6] — source register 1 rt [5:3] — source register 2 imm [5:0] — immediate (I-type, overlaps rt)
Example: 0000_011_001_010_000 → ADD R3, R1, R2  |  0001_001_000_000101 → ADDI R1, R0, 5

alu16.v 
// 16-bit ALU for MiniRISC16
module alu16(
    input  wire [15:0] a,
    input  wire [15:0] b,
    input  wire [2:0]  alu_op,
    output reg  [15:0] result
);
    always @(*) begin
        case(alu_op)
            3'b000: result = a + b;  // ADD
            3'b001: result = a - b;  // SUB
            3'b010: result = a & b;  // AND
            3'b011: result = a | b;  // OR
            3'b100: result = a ^ b;  // XOR
            default: result = 16'b0;
        endcase
    end
endmodule
decoder.v 
module decoder(
    input  wire [15:0] instruction,
    output wire [3:0]  opcode,
    output wire [2:0]  rd, rs, rt,
    output wire [5:0]  imm
);
    assign opcode = instruction[15:12]; // bits 15–12 → opcode
    assign rd     = instruction[11:9];  // bits 11–9  → destination reg
    assign rs     = instruction[8:6];   // bits 8–6   → source reg 1
    assign rt     = instruction[5:3];   // bits 5–3   → source reg 2
    assign imm    = instruction[5:0];   // bits 5–0   → immediate
endmodule
regfile8x16.v 
// 8 registers × 16 bits
module regfile8x16( clk, reg_write, rs, rt, rd, write_data,
                     read_data1, read_data2 );

    reg [15:0] registers [7:0];

    // Combinational reads — instant, no clock needed
    assign read_data1 = registers[rs];
    assign read_data2 = registers[rt];

    // Synchronous write — only on rising clock edge
    always @(posedge clk) begin
        if (reg_write)
            registers[rd] <= write_data;
    end
endmodule

// testing & verification

Each module has its own testbench. The ALU testbench exercises all five operations including a SUB zero case (7 − 7 = 0) to verify the zero flag. The register file testbench checks concurrent read/write behaviour — writing to R4 and R2 while simultaneously reading both ports. The top-level cpu_tb.v runs a full instruction sequence through the assembled datapath and dumps a VCD waveform for analysis in GTKWave.

The test program encoded in instruction memory executes: ADDI R1, R0, 5 → ADDI R2, R0, 3 → ADD R3, R1, R2 → SUB R4, R3, R2, verifying the full fetch-decode-execute path from PC through writeback.
